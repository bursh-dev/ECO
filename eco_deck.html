Why this paper exists

  • Moore's Law has slowed — no more "free" hardware speedups
  • At Google scale, even a 1% gain = a small data center's worth of compute
  • The bottleneck: too much code, too few engineers to hand-optimize
  • What exists today (compilers, Clang-Tidy / lint-style heuristics) leaves a lot on the table
    • Clang-Tidy is basically the C++ analogue of pylint / ruff in our world

ECO in action — one commit, start to finish

  1. Find the slow spot. Google's fleet profiler tracks CPU usage on all production code. ECO looks for hot functions that look like a known anti-pattern.
  2. Write the fix. A fine-tuned LLM proposes a code edit based on the matching recipe.
  3. Check it's safe. Run existing tests → LLM reviews its own work → human code-owner gives final sign-off.
  4. Ship and watch. Merge to production; profiler keeps measuring; auto-revert if anything regresses.

Anti-pattern catalog — 3 of 7 in detail

#1 Unnecessary copy

Before
def process(items):
    items_copy = list(items)
    for x in items_copy:
        log(x)
Allocates a copy of every element
After
def process(items):
    for x in items:
        log(x)
Iterates the original — no copy

#2 Redundant lookup

Before
if config[k] > 0:
    threshold = config[k]
    apply(config[k])
3 hash lookups for the same key
After
v = config[k]
if v > 0:
    threshold = v
    apply(v)
Look up once, reuse the value

#3 Growing a collection

Before
out = []
for x in data:
    out.append(f(x))
Re-allocates as the list grows
After
out = [f(x) for x in data]
# or pre-alloc np.empty
Comp / pre-alloc skips re-allocs

Stage 1 — Learn: build the recipe library

  • Scan ~55,000 historical commits tagged as performance fixes (commit-message keywords, internal speed awards, benchmark deltas)
  • For each, extract the before code and the after code
  • Cluster similar fixes → 7 anti-pattern categories: Alloc, Args, Copy, Map, Move, Sort, Vector
  • Each cluster becomes a "recipe" — a known problem plus concrete examples of how engineers actually fix it

Key idea: ECO doesn't invent optimizations — it industrializes patterns engineers already use.

Stage 2 — Locate: find candidate spots in billions of lines

  • Parse every C++ function with Clang; attach fleet-profiler CPU data → filter to ~10M "costly" functions
    • Clang = the C++ compiler — here used to parse code into a structured tree (AST), so functions can be compared by shape, not just text
    • Fleet-profiler CPU data = Google's always-on profiler that measures CPU time on real production code across the whole fleet (no extra instrumentation needed)
  • Turn each function (and each anti-pattern example) into an embedding — a vector of numbers; similar code → nearby vectors
  • For each anti-pattern, find the top ~500 nearest functions (code-aware embeddings beat text-based ones by ~3×)
  • Re-rank with strict code-similarity metrics (control flow, types) → high-confidence shortlist
  • The LLM is never called blind — only on functions that are hot AND look like a known anti-pattern

This is the strong prior — evolution with where-to-mutate pre-selected.

Stage 3 — Generate: LLM writes the edit

  • Gemini Pro 1.0, fine-tuned on the 55k historical performance commits
  • Four prompting strategies tested:
    • Zero-shot — describe the task, show the code
    • Few-shot — include 2–3 before/after examples in the prompt
    • Chain-of-Thought — ask the model to reason step-by-step before editing
    • ReAct — the model can take actions (read other files, iterate)
  • No single winner — ECO keeps a library, picks strategy per anti-pattern

The most conventional part — same shape as the evolve papers we already know; the engineering is in matching strategy to recipe.

Stage 4 — Verify: gauntlet before & after shipping

  • Run existing CI tests — auto-fix simple build failures; otherwise drop the edit
  • LLM self-review against a checklist (semantics, undefined behavior, side effects)
  • Human code-owner review — same process as a human PR, never skipped
  • Post-deploy monitoring — profiler watches the change in production; auto-revert on regression
  • Net result: >99.5% success, <0.5% revert rate

The part most evolve papers skip — and the reason ECO actually ships.

Stage 1, zoomed in — how the recipe library is built

  • Multiple sources, not just commits:
    • Commits with perf keywords ("speed up", "time/op")
    • Design docs and analysis docs that reference performance-improving code changes
    • Performance tutorials and tips (curated — hand-picked by engineers, not auto-scraped)
    • Prior large-scale refactors (scripted mass rewrites that touched thousands of files — pre-labeled before/after pairs at scale)
    • Commits flagged by analysis tools (e.g., Clang-Tidy)
  • Filter & group: drop reverted / bad commits, cluster the rest
  • Output: a Performance Anti-Pattern Data Set with named categories (explicit_move, string_view, vector_reserve, …)

Click any step for jargon & details.

Jargon for Step 1 — Parse + profile

AST (Abstract Syntax Tree)
A structured representation of code's grammar — what's a function, what's a loop, what's an assignment — stripped of variable names and whitespace. Lets you compare functions by shape, not just text.
Fleet profiler
Google's always-on CPU profiler that measures real production code across the whole fleet of servers. No extra instrumentation needed — the data is already being collected.
Costly functions
Functions that account for a meaningful share of production CPU. Out of all functions in the codebase, only ~10M qualify.
Re-attribution
Pushes CPU cost from generic shared-library calls (like std::vector::push_back) back to the application code that called them. So ECO works on functions you can actually optimize, not on library code shared by millions of callers.

Jargon for Step 2 — Embed & retrieve

Embedding
A list of numbers that represents something (here: a function). Similar code → nearby vectors. The same trick that powers semantic search.
ScaNN
Google's open-source library for fast nearest-neighbor search at scale. Given a query vector, it finds the closest vectors in a huge collection — way faster than comparing all pairs.
3 embedding methods compared
Bag-of-words: just count tokens, ignore order. Deep text embedding: general-purpose text model. Deep code embedding: a model trained to understand code structure — wins by ~3×.
MAP@5
Mean Average Precision at 5. A retrieval-quality metric: out of the top 5 results, how many are actually relevant? Higher = better.

Jargon for Step 3 — Re-rank

BLEU
A text-similarity metric originally from machine translation. Measures how much the n-grams (word/token sequences) of two texts overlap. Used here to compare code structurally.
ROUGE-L
Another text-similarity metric. Focuses on the longest common subsequence between two texts — sensitive to the order of tokens, not just their counts.
Type-set intersection
Do the two functions use the same set of C++ types? E.g., both use std::string and std::vector<int>? Strong signal that they're doing similar things.
Control-flow similarity
Do the loops, branches, and returns line up between the two functions? Sensitive to the shape of the logic, not its surface text.
Why strip names & comments first
The structural shape of the code is what matters — not the coincidental fact that one engineer named a variable foo and another named it bar.

Result: a small, high-confidence shortlist per anti-pattern → sent to Stage 3 (LLM).

Stage 3, zoomed in — "Code Edit Generation Strategies"

Fine-tuned Gemini Pro 1.0 on the 55k perf commits — then tried 4 prompting strategies:

Zero-shot

Describe the task + show the code. No examples, no reasoning.

Cheapest. Strong baseline.

Few-shot

Include 2–3 before/after examples in the prompt.

Useful when the pattern is subtle.

Chain-of-Thought

Ask the model to reason step-by-step before editing.

More diverse edits, higher invalid rate.

ReAct

Model takes actions (cat, patch), iterates.

Best when context matters.

Key finding: no single strategy wins everywhere.
  • Zero-shot & ReAct lead overall
  • Different anti-patterns prefer different strategies (Copy ↔ Zero-shot; Vector ↔ ReAct)
  • ECO keeps a library of strategies and picks per recipe

Why this paper exists

  • Moore's Law has slowed — no more "free" hardware speedups
  • At Google scale, even a 1% gain = a small data center's worth of compute
  • The bottleneck: too much code, too few engineers to hand-optimize

ECO in action — one commit, start to finish

  1. Find the slow spot. Google's fleet profiler tracks CPU usage on all production code. ECO looks for hot functions that look like a known anti-pattern.
  2. Write the fix. A fine-tuned LLM proposes a code edit based on the matching recipe.
  3. Check it's safe. Run existing tests → LLM reviews its own work → human code-owner gives final sign-off.
  4. Ship and watch. Merge to production; profiler keeps measuring; auto-revert if anything regresses.

#1 Unnecessary copy

Before
def process(items):
    items_copy = list(items)
    for x in items_copy:
        log(x)
Allocates a copy of every element
After
def process(items):
    for x in items:
        log(x)
Iterates the original — no copy

#2 Redundant lookup

Before
if config[k] > 0:
    threshold = config[k]
    apply(config[k])
3 hash lookups for the same key
After
v = config[k]
if v > 0:
    threshold = v
    apply(v)
Look up once, reuse the value

#3 Growing a collection

Before
out = []
for x in data:
    out.append(f(x))
Re-allocates as the list grows
After
out = [f(x) for x in data]

Comp / pre-alloc skips re-allocs

"Move" anti-pattern — transfer ownership instead of copying

What it is in C++:

C++ has move semantics: instead of copying a big object (which allocates new memory and duplicates contents), you can move it — transfer ownership of the underlying memory to a new variable without copying. The original is left in a "moved-from" state and the new variable just inherits the existing memory.

The anti-pattern: writing code that copies when it could move — especially for large objects like strings, vectors, maps that own heap memory.

// Before — copies the whole string
std::string s = build_huge_string();
process(s);  // s passed by value → full copy made

// After — moves it, no copy
std::string s = build_huge_string();
process(std::move(s));  // ownership transferred, no allocation

Python analogue: Python doesn't have explicit move semantics. Variables in Python are references to objects, so assignment never copies by default (which is why Python doesn't have this category of bug). The closest analogue: dropping unnecessary copy.deepcopy() / list(x) / dict(x) calls when you don't actually need an independent copy.

"Vector" anti-pattern — pre-size your container

What std::vector is in C++:

A resizable array container — basically the C++ equivalent of a Python list. Grows by doubling its underlying memory block; every doubling triggers an allocation + bulk copy of all existing elements.

The anti-pattern: building up a vector with push_back in a loop without telling it the size first. Triggers a cascade of reallocations.

// Before — ~14 reallocations + ~10k cumulative copies
std::vector<int> results;
for (int i = 0; i < 10000; i++)
  results.push_back(compute(i));

// After — one-line fix: pre-allocate
std::vector<int> results;
results.reserve(10000);  // ← entire fix
for (int i = 0; i < 10000; i++)
  results.push_back(compute(i));

Python analogues:

  • List comprehensions over .append() loops — tighter and often pre-sized internally
  • NumPy: use np.empty(n) + index assignment instead of np.append in a loop — NumPy's append reallocates every time, which is much worse than Python lists or C++ vectors

Stage 1 — Learn: build the recipe library (offline / one-time)

  • Multiple sources, not just commits:
    • Commits with perf keywords ("speed up", "time/op")
    • Design docs and analysis docs that reference performance-improving code changes
    • Performance tutorials and tips (curated — hand-picked by engineers, not auto-scraped)
    • Prior large-scale refactors (scripted mass rewrites that touched thousands of files — pre-labeled before/after pairs at scale)
    • Commits flagged by analysis tools (e.g., Clang-Tidy)
  • Filter & group: drop reverted / bad commits, cluster the rest
  • Output: a Performance Anti-Pattern Data Set with named categories (explicit_move, string_view, vector_reserve, …)

Runs once / offline — feeds the online loop on the next slide. ECO doesn't invent optimizations — it industrializes patterns engineers already use.

Stage 2 — Locate: find candidate spots in billions of lines

  • Parse every C++ function with Clang; attach fleet-profiler CPU data → filter to ~10M "costly" functions
    • Clang = the C++ compiler — here used to parse code into a structured tree (AST), so functions can be compared by shape, not just text
    • Fleet-profiler CPU data = Google's always-on profiler — Google-Wide Profiling (GWP) — that samples CPU time on real production code across the whole fleet (no extra instrumentation needed)
  • Turn each function (and each anti-pattern example) into an embedding — a vector of numbers; similar code → nearby vectors
  • For each anti-pattern, find the top ~500 nearest functions (code-aware embeddings beat text-based ones by ~3×)
  • Re-rank with strict code-similarity metrics → a smaller, high-confidence shortlist per anti-pattern (paper doesn't give the exact cutoff)
  • The LLM is never called blind — only on functions that are hot AND look like a known anti-pattern

This is the strong prior — evolution with where-to-mutate pre-selected.

Stage 3 — Generate: LLM writes the edit

  • Gemini Pro 1.0, fine-tuned on the 55k historical performance commits
  • Four prompting strategies tested:
    • Zero-shot — describe the task, show the code
    • Few-shot — include 2–3 before/after examples in the prompt
    • Chain-of-Thought — ask the model to reason step-by-step before editing
    • ReAct — the model can take actions (read other files, iterate)
  • No single winner — ECO keeps a library, picks strategy per anti-pattern

The most conventional part — same shape as the evolve papers we already know; the engineering is in matching strategy to recipe.

Stage 4 — Verify: gauntlet before & after shipping

  • Run existing CI tests — auto-fix simple build failures; otherwise drop the edit
  • LLM self-review against a checklist (semantics, undefined behavior, side effects)
  • Human code-owner review — same process as a human PR, never skipped
  • Post-deploy monitoring — profiler watches the change in production; auto-revert on regression
  • Net result: >99.5% success, <0.5% revert rate

The part most evolve papers skip — and the reason ECO actually ships.

The "where in billions of lines" problem, in 3 sub-steps:

1

Parse + profile

  • Attach fleet profiler CPU data
  • Filter to ~10M "costly" functions
  • Re-attribution pushes cost from shared-lib calls back to app code — so you optimize where you can actually edit
→
2

Embed & retrieve

  • Each function → vector (embedding)
  • For each anti-pattern, find top ~500 nearest via ScaNN
  • 3 embedding methods tested:
    • Bag-of-words (count tokens, ignore order)
    • Deep text embedding (general-purpose text model)
    • Deep code embedding — winner (~3× better MAP@5)
  • Sharper query: match on the before→after diff, not just the anti-pattern shape → MAP@5 jumps from 0.20 to 0.54

3

Re-rank

  • Composite score across:
    • BLEU + ROUGE-L
    • Type-set intersection
    • Control-flow similarity
  • Strips variable names & comments before comparing — focuses on structure
  • Conservative matches preferred

Jargon for Step 1 — Parse + profile

AST (Abstract Syntax Tree) (click for example)
Code parsed into a tree — functions, loops, assignments — stripped of names and whitespace. Compare by shape, not text.
Fleet profiler — aka Google-Wide Profiling (GWP) (click for details)
Google's always-on CPU profiler across all production servers. Samples real workloads continuously; the data is already collected, ECO just reuses it.
Costly functions
Functions that account for a meaningful share of production CPU. ~10M qualify out of the whole codebase.
Re-attribution
Pushes CPU cost from shared-lib calls (like std::vector::push_back) back to the app code that called them — so you optimize where you can actually edit.

Jargon for Step 2 — Embed & retrieve

Embedding + ScaNN (click for details)
Embedding = a list of numbers representing a function; similar code → nearby vectors (same trick as semantic search). ScaNN = Google's open-source library for fast nearest-neighbor search at scale — finds the closest vectors without comparing every pair.
Deep code embedding (the winner)
A 1B-param dual-encoder (Google-internal, name not disclosed) — fine-tuned on ~25M code pairs (similar-function pairs + diff/before pairs).
MAP@5 (click for analogy: top 500 vs. top 5)
Mean Average Precision at 5. Retrieval-quality metric: out of the top 5 results, how many are relevant and well-ranked? Higher = better. Paper's numbers: bag-of-words 0.07, deep text 0.10, deep code 0.20, deep code + diff queries 0.54.

Jargon for Step 3 — Re-rank

BLEU & ROUGE-L
Text-similarity metrics from machine translation. BLEU = n-gram (token-sequence) overlap. ROUGE-L = longest common subsequence (sensitive to order). Used here on code.
Type-set intersection
Do the two functions use the same C++ types (e.g. both use std::string and std::vector<int>)? Strong signal they're doing similar things.
Control-flow similarity
Do the loops, branches, and returns line up between the two functions? Sensitive to the shape of the logic, not its surface text.
Why strip names & comments first
Focus on structural shape, not coincidental naming (one engineer's foo is another's bar).

Result: a small, high-confidence shortlist per anti-pattern → sent to Stage 3 (LLM).

AST example — Python

Python source code:

def add(a, b):
    return a + b

Its AST (the structured tree) looks like this:

Module └── FunctionDef name='add' ├── args │ ├── arg name='a' │ └── arg name='b' └── body └── Return └── BinOp ├── left: Name id='a' ├── op: Add └── right: Name id='b'

Each node is a grammatical construct. Names and whitespace are stripped — what matters is the shape. Two functions with different names but the same shape have identical ASTs.

Google-Wide Profiling (GWP)

What it is:

  • Always-on CPU profiler running on every Google production server (since ~2010)
  • Statistical sampling — periodic ~10ms snapshots of each process's CPU activity, no manual instrumentation needed
  • Centrally aggregates samples into a fleet-wide view: "function foo() consumed 2.4% of service X's CPU this week"
  • Originally a 2010 Google Research paper; gave us the open-source pprof profile format

Open-source equivalents you can play with:

  • py-spy — sampling profiler for a single Python process
  • Pyroscope — open-source continuous profiler for many services (closest analogue)
  • Datadog Continuous Profiler, Grafana Pyroscope, Polar Signals — commercial SaaS versions

How ECO uses it (twice):

  • Stage 2 (Locate): "which functions are worth optimizing?" — the hot ones
  • Stage 4 (Verify): "did our edit actually make things faster in production?" — auto-revert if not

How ECO uses Embedding + ScaNN together

Once (offline, when the codebase is scanned):

For each of 10M functions in the codebase: embed it → vector Build a ScaNN index over those 10M vectors.

Continuously (online):

For each of the 7 anti-patterns: For each example of the pattern (the "before" code): embed it → query vector scann.search(query, top_k=500) ← finds 500 similar functions in ms send those 500 to the re-ranking step

That's the whole Embed + ScaNN machinery in eight conceptual lines.

In one sentence: Embedding = how to turn code into numbers so "similar code" becomes "nearby vectors." ScaNN = how to find the nearest neighbors among millions of vectors fast enough to do this continuously across Google's fleet.

MAP@5 — why "5" when production uses "500"?

Think of it like restaurant reviews. A restaurant serves 500 dishes a day, but a critic only orders 5 items to evaluate the menu. The critic's score isn't saying "only 5 dishes are good" — it's saying "the top of the menu represents what you'd actually order most often, so let's check whether that is any good."

Same idea here:

  • Top 500 = how many candidates ECO's production system actually keeps and sends downstream.
  • MAP@5 = how the team evaluates whether the ranking is any good — by inspecting just the very top.

Why measure at 5, not 500?

  • The top of the ranking matters most. If the top 5 is garbage, the broader ranking is probably bad too.
  • Small K (like 5) is more sensitive to ranking-quality differences. MAP@500 would average out into mush.
  • It's the standard convention in information retrieval — you'll see MAP@5 / @10 / @100 throughout the literature.

Stage 3, zoomed in — "Code Edit Generation Strategies"

Fine-tuned Gemini Pro 1.0 on the 55k perf commits — then tried 4 prompting strategies:

Zero-shot

Describe the task + show the code. No examples, no reasoning.

Cheapest. Strong baseline.

Few-shot

Include 2–3 before/after examples in the prompt.

Useful when the pattern is subtle.

Chain-of-Thought

Ask the model to reason step-by-step before editing.

More diverse edits, higher invalid rate.

ReAct

Model takes actions (cat, patch), iterates.

Best when context matters.

Key finding: no single strategy wins everywhere.
  • Zero-shot & ReAct lead overall — on a held-out test of 48 real Google commits, both score ~0.5 / 1.0 on human quality reviews (CoT trails at ~0.34) (click for details)
  • Different anti-patterns prefer different strategies (Copy ↔ Zero-shot; Vector ↔ ReAct)
  • ECO keeps a library of strategies and picks per recipe

How they tested:

"Held-out test of 48 real Google commits" means:

  • Took 48 actual performance-improving commits from Google's codebase
  • Each one has the original slow code (before) and the engineer's fix (after)
  • The model was never shown these 48 during training — that's what "held-out" means

For each commit: give the LLM the "before" code, ask it to produce a fix, then compare the LLM's fix against the human's.

"Human quality reviews" = a person (likely a Google engineer) read both versions and scored the LLM's output on a 0–1 scale:

  • 0 = bad / wrong / nonsense
  • 1 = matches the human's fix in quality (or better)
